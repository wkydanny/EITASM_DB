Class Diagram and ERD

Step-by-step lecture and lab for beginner AEC students. This document explains what class diagrams and ERDs are, why we use them, how they are different, and how to build them from a real-life system.

Learning Objectives

  • Explain what a class diagram is.
  • Explain what an ERD is.
  • Identify classes, entities, attributes, methods, relationships, primary keys, and foreign keys.
  • Create a simple class diagram from requirements.
  • Create a simple ERD from the same requirements.
  • Understand the difference between software design and database design.

1. Definitions

A class diagram is a UML diagram used in software design. It shows the main classes in the application, their attributes, their methods, and their relationships.

Simple: class diagram = how the application code is organized.

Example: User, Account, Transaction, Category.

An ERD means Entity Relationship Diagram. It is used in database design. It shows tables, columns, primary keys, foreign keys, and relationships.

Simple: ERD = how the database is structured.

Example: Users, Accounts, Transactions, Categories.

Class DiagramERD
Software/code designDatabase design
Shows classesShows tables/entities
Has attributes and methodsHas columns, PK, FK
Used by programmersUsed by database developers
User classUsers table
Blunt rule: Class diagram = code structure. ERD = database structure. Related, but not the same.

3. Example Scenario

We will use a Finance Tracker App. Users create accounts, categories, and transactions.

  • A user can have many accounts.
  • A user can create many categories.
  • An account can have many transactions.
  • A category can be used by many transactions.
  • Each transaction has amount, date, type, and description.

4. Build a Class Diagram

Step 1 — Read requirements

Do not draw blindly. Read the business case first.

Step 2 — Find nouns

Nouns often become classes: User, Account, Category, Transaction.

Step 3 — Find attributes
User: UserID, FullName, Email, PasswordHash
Account: AccountID, AccountName, AccountType, Balance
Category: CategoryID, CategoryName, CategoryType
Transaction: TransactionID, Amount, TransactionDate, TransactionType, Description
Step 4 — Find methods
User: Login(), UpdateProfile()
Account: UpdateBalance(), GetTransactions()
Category: RenameCategory()
Transaction: CalculateSignedAmount(), ValidateTransaction()
Step 5 — Add relationships
  • One User has many Accounts.
  • One User has many Categories.
  • One Account has many Transactions.
  • One Category has many Transactions.
Step 6 — Draw it

5. Build an ERD

Step 1 — Use the same requirements

The same system becomes database tables.

Step 2 — Find entities

Entities usually become tables: Users, Accounts, Categories, Transactions.

Step 3 — Add columns

Each table needs columns that describe the entity.

Step 4 — Choose primary keys
Users: UserID
Accounts: AccountID
Categories: CategoryID
Transactions: TransactionID
Step 5 — Add foreign keys
Accounts.UserID -> Users.UserID
Categories.UserID -> Users.UserID
Transactions.AccountID -> Accounts.AccountID
Transactions.CategoryID -> Categories.CategoryID
Step 6 — Draw the ERD

6. SQL Schema from the ERD

CREATE TABLE Users (
    UserID INT IDENTITY(1,1) PRIMARY KEY,
    FullName VARCHAR(100) NOT NULL,
    Email VARCHAR(150) NOT NULL UNIQUE,
    PasswordHash VARCHAR(255) NOT NULL,
    CreatedDate DATETIME DEFAULT GETDATE()
);

CREATE TABLE Accounts (
    AccountID INT IDENTITY(1,1) PRIMARY KEY,
    UserID INT NOT NULL,
    AccountName VARCHAR(100) NOT NULL,
    AccountType VARCHAR(50) NOT NULL,
    Balance DECIMAL(10,2) DEFAULT 0,
    FOREIGN KEY (UserID) REFERENCES Users(UserID)
);

CREATE TABLE Categories (
    CategoryID INT IDENTITY(1,1) PRIMARY KEY,
    UserID INT NOT NULL,
    CategoryName VARCHAR(100) NOT NULL,
    CategoryType VARCHAR(20) NOT NULL,
    FOREIGN KEY (UserID) REFERENCES Users(UserID)
);

CREATE TABLE Transactions (
    TransactionID INT IDENTITY(1,1) PRIMARY KEY,
    AccountID INT NOT NULL,
    CategoryID INT NOT NULL,
    Amount DECIMAL(10,2) NOT NULL,
    TransactionDate DATE NOT NULL,
    TransactionType VARCHAR(20) NOT NULL,
    Description VARCHAR(255),
    FOREIGN KEY (AccountID) REFERENCES Accounts(AccountID),
    FOREIGN KEY (CategoryID) REFERENCES Categories(CategoryID)
);
Important: The ERD is not decoration. It should help you write the real SQL schema.

8. Relationship Types

One-to-One

One record connects to one record.

Users 1 --- 1 Profiles
One-to-Many

One record connects to many records.

Users 1 --- * Accounts
Many-to-Many

Many records connect to many records. Needs a junction table.

Students * --- * Courses
Many-to-Many SQL Example
CREATE TABLE Students (StudentID INT PRIMARY KEY, StudentName VARCHAR(100));
CREATE TABLE Courses (CourseID INT PRIMARY KEY, CourseName VARCHAR(100));

CREATE TABLE StudentCourses (
    StudentID INT,
    CourseID INT,
    PRIMARY KEY (StudentID, CourseID),
    FOREIGN KEY (StudentID) REFERENCES Students(StudentID),
    FOREIGN KEY (CourseID) REFERENCES Courses(CourseID)
);

9. Step-by-Step Tutorial: Online Store

Scenario: A small online store sells products. Customers place orders. Each order can contain many products. Products belong to categories.

Step 1 — Extract nouns
  • Customer
  • Order
  • Product
  • Category
  • OrderItem
Step 2 — Class diagram draft
Customer: CustomerID, FullName, Email, PlaceOrder()
Order: OrderID, OrderDate, Status, CalculateTotal()
Product: ProductID, ProductName, Price, UpdatePrice()
Category: CategoryID, CategoryName
OrderItem: Quantity, UnitPrice, CalculateLineTotal()
Step 3 — ERD draft
Customers(CustomerID PK, FullName, Email)
Orders(OrderID PK, CustomerID FK, OrderDate, Status)
Categories(CategoryID PK, CategoryName)
Products(ProductID PK, CategoryID FK, ProductName, Price)
OrderItems(OrderID FK, ProductID FK, Quantity, UnitPrice)
Step 4 — SQL schema
CREATE TABLE Customers (
    CustomerID INT IDENTITY(1,1) PRIMARY KEY,
    FullName VARCHAR(100) NOT NULL,
    Email VARCHAR(150) NOT NULL UNIQUE
);

CREATE TABLE Orders (
    OrderID INT IDENTITY(1,1) PRIMARY KEY,
    CustomerID INT NOT NULL,
    OrderDate DATE NOT NULL,
    Status VARCHAR(50) NOT NULL,
    FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
);

CREATE TABLE Categories (
    CategoryID INT IDENTITY(1,1) PRIMARY KEY,
    CategoryName VARCHAR(100) NOT NULL
);

CREATE TABLE Products (
    ProductID INT IDENTITY(1,1) PRIMARY KEY,
    CategoryID INT NOT NULL,
    ProductName VARCHAR(100) NOT NULL,
    Price DECIMAL(10,2) NOT NULL,
    FOREIGN KEY (CategoryID) REFERENCES Categories(CategoryID)
);

CREATE TABLE OrderItems (
    OrderID INT NOT NULL,
    ProductID INT NOT NULL,
    Quantity INT NOT NULL,
    UnitPrice DECIMAL(10,2) NOT NULL,
    PRIMARY KEY (OrderID, ProductID),
    FOREIGN KEY (OrderID) REFERENCES Orders(OrderID),
    FOREIGN KEY (ProductID) REFERENCES Products(ProductID)
);

10. Common Mistakes

MistakeWhy wrongCorrect
No primary keyNo unique row identifier.Add ID as PK.
FK in wrong tableRelationship becomes confusing.Put FK on the many side.
Methods in ERDMethods belong to code.Only columns in ERD.
Many-to-many without junction tableCannot store it cleanly.Create junction table.
Vague namesHard to understand.Use clear names like Customers, Orders.

11. Student Lab Instructions

Create a class diagram and ERD for one system:

  • Ice cream ordering app
  • Gym membership system
  • Library management system
  • Student course registration system
  • Small product/service business system
Required Steps
  1. Write a short paragraph describing your system.
  2. Extract the main nouns.
  3. Convert nouns into classes.
  4. Add attributes to every class.
  5. Add at least one method to important classes.
  6. Draw the class diagram.
  7. Convert classes into database tables.
  8. Add primary keys.
  9. Add foreign keys.
  10. Show relationships and cardinality.
  11. Write the SQL schema.
Minimum Requirements
  • At least 4 classes.
  • At least 4 tables.
  • At least 3 relationships.
  • At least 3 primary keys.
  • At least 2 foreign keys.
  • At least one one-to-many relationship.
  • If many-to-many exists, create a junction table.
